# ContextPack 生命周期机制

## 1. 核心原则

ContextPack 不是后期一次性生成的报告。

它应该从项目开始就建立，并随着任务持续更新。

最小原则：

> 项目开始时建立 L1；  
> 重要任务后记录 Context Delta；  
> 阶段性整理为 L2 / L3。

## 2. 为什么不能等后期一次性生成

如果项目一开始没有 ContextPack：

- 每次 AI 都要重新理解项目；
- 聊天上下文容易丢；
- 模块边界和当前状态不稳定；
- 重要决策没有沉淀；
- 后续工具生成的上下文可能缺少历史原因。

工具可以后期补全事实，但不能替代项目过程中的持续记录。

## 3. Bootstrap：项目启动时创建 L1

项目开始或第一次接入本套件时，应建立 L1 ContextPack。

L1 不需要很复杂，只要包含：

- 项目定位
- 当前阶段
- 技术栈
- 主要目录
- 常用命令
- 当前重点
- 禁止事项
- 关键模块入口
- 已知风险
- 最近任务状态

这些可以来自：

- `PROJECT_INTAKE`
- README
- package.json
- 目录结构
- 用户描述
- 项目负责人判断
- CLI / 插件扫描

## 4. Delta：每个重要任务后的上下文增量

重要任务完成后，应记录 Context Delta。

Context Delta 至少包括：

- 本次完成了什么
- 哪些模块发生变化
- 哪些规则发生变化
- 哪些命令发生变化
- 哪些风险新增或消失
- 哪些文档需要回写
- 下一步重点是什么

## 5. Writeback：如何写入 ContextPack

如果 CommandPack 允许修改 ContextPack，应直接更新。

如果不允许，应在 ExecutionReport 中写：

- 需要更新的 ContextPack 层级
- 建议补充内容
- 为什么需要更新
- 是否阻断下一轮任务

## 6. L1 / L2 / L3 拆分结构

默认推荐结构：

- `docs/CONTEXT_PACK.md`：上下文入口 + L1 最小可用上下文
- `docs/context/CONTEXT_PACK_L2.md`：深入协作上下文
- `docs/context/CONTEXT_PACK_L3.md`：审计 / 交接上下文

L1 / L2 / L3 是上下文密度层级，不只是文件名。

普通任务只需要 L1；
中等复杂任务读取 L1 + L2；
高风险、架构、权限、计费、数据库、迁移、发布任务读取 L1 + L2 + L3。

小项目仍可使用单文件模式，但长期项目建议采用拆分结构。

### L1：最小可用上下文

用于普通任务。位于 `docs/CONTEXT_PACK.md`，是 AI 进入项目时的上下文入口。

### L2：深入协作上下文

位于 `docs/context/CONTEXT_PACK_L2.md`。

当项目开始跨模块、多人协作、长期迭代时，补充：

- 模块关系
- 数据结构
- 关键流程
- 已知风险
- 最近 PR 摘要

### L3：审计交接上下文

位于 `docs/context/CONTEXT_PACK_L3.md`。

当项目进入发布、重构、迁移、审计阶段时，补充：

- 架构决策
- 历史原因
- 风险清单
- 回滚策略
- 发布状态
- 长期计划

## 7. 普通用户不需要手写 ContextPack

ContextPack 不要求普通用户手写。项目启动时可以先通过 `PROJECT_INTAKE` 生成 L1，后续由项目负责人、规划型 AI、CLI、插件或执行代理在授权范围内维护。

---

## 8. 谁负责更新

可以由以下角色更新：

- 项目负责人
- 规划型 AI
- 代码执行代理，在 CommandPack 授权时
- CLI / 插件
- 后续 RepoSense 等分析工具

关键不是谁写，而是必须形成可追踪的上下文记录。

## 9. 任务开始前的规则

任务开始前应判断：

- 是否已有 ContextPack
- L1 是否足够支撑本次任务
- 是否需要先 Bootstrap ContextPack
- 是否需要读取项目资料
- 是否需要在任务结束后回写上下文

如果没有 ContextPack，但任务复杂度超过小修，应优先建立 L1。

## 10. 一句话

ContextPack 不是一次性文档。  
它是项目的长期工作记忆。
