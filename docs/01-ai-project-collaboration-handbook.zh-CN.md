# AI 编程项目协作手册 v1.0  
## 面向项目负责人的项目启动与持续开发规范

## 0. 核心理念

不要一开始就让 AI 写代码。

一个项目真正稳定，不是因为 AI 能力强，而是因为项目一开始就有：

- 清楚的目标
- 清楚的边界
- 清楚的模块
- 清楚的文档
- 清楚的任务合同
- 清楚的测试门禁
- 清楚的回滚方式

AI 适合做执行者，不适合在上下文混乱、边界模糊、规则缺失的仓库里自由发挥。

所以项目协作的基本原则是：

> 人负责方向、边界、判断和验收；  
> AI 负责在明确任务合同下执行；  
> 仓库文档负责保存长期上下文；  
> 测试和门禁负责验证结果。

---

## 1. 项目开始前：先不要写代码

项目启动时，第一步不是建页面、写功能、调 UI，而是先把项目的“骨架规则”立起来。

至少要先回答这些问题：

1. 这个项目解决什么问题？
2. 用户是谁？
3. MVP 包含哪些功能？
4. 哪些功能暂时不做？
5. 数据结构大概是什么？
6. 核心模块有哪些？
7. 哪些文件是入口？
8. 哪些文件未来不能继续膨胀？
9. 测试和构建怎么跑？
10. 代码出了问题怎么回滚？

如果这些问题没有回答清楚，直接让 AI 写代码，后面很容易变成：

- 功能堆功能
- 文件越来越大
- 逻辑到处复制
- 中文文案乱码
- 没有测试
- 没法回滚
- AI 每次都重新猜项目结构

---

## 2. 项目初始化必须建立的基础文档

一个新项目，建议一开始至少建立以下文档：

```txt
repo/
  README.md
  AGENTS.md
  docs/
    PRODUCT_BRIEF.md
    PRODUCT_ARCHITECTURE.md
    MODULE_BOUNDARY.md
    DEV_GUIDE.md
    TESTING.md
    CONTEXT_PACK.md
    PR_SUMMARIES.md
  scripts/
    check_utf8.py
  .editorconfig
  .gitattributes
  .gitignore
```

这些文件的作用不同，不要混在一起。

---

## 3. README.md：项目对外说明

`README.md` 负责说明这个项目是什么。

应该包括：

- 项目名称
- 项目定位
- 当前版本状态
- 核心功能
- 技术栈
- 本地启动方式
- 构建方式
- 测试方式
- 当前不做什么

README 不需要写太多内部细节。  
它的重点是让任何人打开仓库后，能快速知道：

> 这个项目是什么，怎么跑，现在做到哪一步。

---

## 4. AGENTS.md：给 AI 执行代理看的仓库规则

`AGENTS.md` 是项目级 AI 协作规则。

它不是产品说明书，而是告诉 Codex / IDE coding agent / CLI coding agent：

- 这个仓库的结构是什么。
- 哪些目录可以改。
- 哪些目录不能乱改。
- 运行命令是什么。
- 编码规则是什么。
- 测试门禁是什么。
- 哪些文件不能继续变大。
- 哪些行为禁止。
- 完成后必须怎么报告。

建议 `AGENTS.md` 包含：

```md
# AGENTS.md

## 项目概况

一句话说明项目是什么。

## 仓库结构

说明主要目录用途。

## 常用命令

- 安装依赖：
- 本地开发：
- 测试：
- 构建：
- UTF-8 检查：

## 修改规则

- 不得顺手重构。
- 不得扩大范围。
- 不得修改密钥和生产配置。
- 不得提交构建产物。
- 不得破坏中文 UTF-8。

## 模块边界

说明哪些文件只做入口，哪些目录承载业务逻辑。

## 门禁要求

说明任务完成后必须运行哪些检查。

## ExecutionReport 要求

说明完成后必须报告什么。
```

`AGENTS.md` 是让 AI 稳定执行的第一层项目约束。

---

## 5. PRODUCT_BRIEF.md：产品意图文档

`docs/PRODUCT_BRIEF.md` 负责回答：

- 这个产品为什么存在？
- 目标用户是谁？
- 核心痛点是什么？
- MVP 做到什么程度？
- 当前版本不做什么？
- 后续可能扩展什么？

这个文件防止项目在开发过程中不断漂移。

一个产品最怕的是：  
今天觉得这个功能有用就加，明天觉得那个也不错也加，最后变成什么都想做，但什么都不稳。

`PRODUCT_BRIEF.md` 的作用就是提醒项目：

> 先把核心闭环做稳，不要乱扩。

---

## 6. PRODUCT_ARCHITECTURE.md：产品与系统架构

`docs/PRODUCT_ARCHITECTURE.md` 负责说明系统结构。

应该包括：

- 核心业务流程。
- 核心数据结构。
- 核心状态流转。
- 前端、后端、本地存储、AI 调用之间的关系。
- 重要 API contract。
- 关键模块职责。
- 未来扩展点。

开发前不一定要写得特别完整，但至少要有一个初版。

尤其是涉及：

- 用户权限
- 套餐
- license
- 导入导出
- 本地文件写入
- AI 生成
- 支付
- 数据同步

这些必须先有结构说明，再让 AI 执行。

---

## 7. MODULE_BOUNDARY.md：模块边界文档

`docs/MODULE_BOUNDARY.md` 是防止屎山代码的关键文档。

它负责说明：

- 哪些模块负责什么。
- 哪些文件只是入口。
- 哪些文件不能继续堆逻辑。
- 新功能应该放在哪里。
- 旧大文件未来如何逐步拆分。
- 哪些逻辑禁止跨层调用。

例如：

```md
## 中央入口文件规则

以下文件只允许做 wiring / 初始化 / 路由接线：

- main.js
- index.js
- App.tsx
- preload.ts
- renderer.ts

不得继续向这些文件加入大段业务逻辑。

新功能应优先新增：

- services/
- schemas/
- parsers/
- builders/
- views/
- components/
- stores/
- validators/
- exporters/
```

这个文件非常重要。

很多 AI 不是故意写屎山，而是在没有边界的情况下，为了完成任务，把所有逻辑都塞进眼前能改的文件里。

---

## 8. DEV_GUIDE.md：开发指南

`docs/DEV_GUIDE.md` 负责说明开发约定。

包括：

- 技术栈。
- 命名规范。
- 文件组织方式。
- TypeScript 规则。
- UI 组件约定。
- 状态管理方式。
- 错误处理方式。
- 日志规则。
- 本地开发注意事项。

它解决的是：

> 代码应该按什么风格写。

---

## 9. TESTING.md：测试与门禁文档

`docs/TESTING.md` 负责说明项目怎么验证。

至少包括：

- lint 命令。
- typecheck 命令。
- test 命令。
- build 命令。
- UTF-8 检查命令。
- smoke test 命令。
- 哪些失败可以接受。
- 哪些失败必须阻断。

例如：

```md
## 必跑门禁

普通 PR：

- npm run lint
- npm run typecheck
- npm test
- npm run build

涉及中文文案：

- python scripts/check_utf8.py

涉及导出功能：

- npm run test:export
- 手动检查导出文件
```

没有测试门禁，AI 很容易“感觉完成了”。  
有测试门禁，项目才有现实锚点。

---

## 10. CONTEXT_PACK.md：仓库上下文包

`docs/CONTEXT_PACK.md` 是给 AI 快速理解项目的上下文包。

它不是所有文档的复制，而是项目当前状态的压缩版。

建议分三层：

### L1：最小可用上下文

用于普通小任务。

包括：

- 项目定位。
- 当前版本。
- 主要目录。
- 常用命令。
- 当前重点。
- 禁止事项。
- 关键模块入口。

### L2：深入协作上下文

用于跨模块任务。

包括：

- 主要业务流程。
- 数据结构。
- 模块依赖关系。
- 已知技术债。
- 当前架构边界。
- 最近 PR 总结。

### L3：审计交接上下文

用于重大重构、迁移、长期交接。

包括：

- 完整架构说明。
- 历史决策。
- 风险点。
- 回滚策略。
- 已完成阶段。
- 未完成阶段。
- 下一阶段计划。

ContextPack 的目标是：

> 不要每次都在聊天里重新解释项目，而是让仓库自己携带上下文。

---

## 11. PR_SUMMARIES.md：每次迭代沉淀

`docs/PR_SUMMARIES.md` 用来记录每次重要变更。

每次 PR 完成后，应该追加：

```md
## PR-XXX：标题

- 日期：
- 目标：
- 修改文件：
- 核心实现：
- 执行门禁：
- 风险：
- 回滚方式：
- 下一步：
```

这个文件的作用很大。

它能让后续 AI 快速知道：

- 之前做过什么。
- 为什么这样做。
- 哪些地方不能乱动。
- 下一步应该接着哪里走。

没有 PR_SUMMARIES，项目会越来越依赖聊天记录。  
而聊天记录是不稳定的，仓库文档才是长期上下文。

---

## 12. UTF-8 与中文防乱码基础配置

任何含中文的项目，必须从一开始就建立 UTF-8 规则。

建议添加 `.editorconfig`：

```ini
root = true

[*]
charset = utf-8
end_of_line = lf
insert_final_newline = true
trim_trailing_whitespace = true
```

建议添加 `.gitattributes`：

```gitattributes
* text=auto eol=lf
*.md text working-tree-encoding=UTF-8
*.json text working-tree-encoding=UTF-8
*.js text working-tree-encoding=UTF-8
*.ts text working-tree-encoding=UTF-8
*.tsx text working-tree-encoding=UTF-8
*.html text working-tree-encoding=UTF-8
*.css text working-tree-encoding=UTF-8
```

建议添加：

```txt
scripts/check_utf8.py
```

用于检查：

- 文件是否能按 UTF-8 读取。
- 是否出现疑似乱码字符。
- 是否有明显 mojibake。

中文项目必须把 UTF-8 当作硬门禁，而不是出了问题再补救。

## 13. Git 初始化与可回滚纪律

项目初始化后，应尽早建立 Git 基线。

推荐流程：

```bash
git init
git add .
git commit -m "chore: initial project baseline"
git tag baseline-v0
```

每个阶段都应该保持：

- 一次迭代一个意图。
- 一个 PR 一个目标。
- 提交前检查 diff。
- 不提交密钥。
- 不提交构建产物。
- 不提交 node_modules。
- 不提交乱码文件。

建议 `.gitignore` 至少包含：

```gitignore
node_modules/
dist/
build/
.env
.env.local
*.log
.cache/
.DS_Store
```

重要节点建议打 tag：

```bash
git tag before-license-integration
git tag before-ui-refactor
git tag v0.1.0-beta
```

这样出问题时可以回滚。

## 14. 项目开发过程：每次任务怎么做

每次让 AI 执行前，不要直接说：

> 帮我把这个功能做了。

而是先形成一个明确的任务合同，也就是 CommandPack。

对于不熟悉工程规范的用户，不要求直接手写 CommandPack。
可以先填写 `templates/TASK_INTAKE.md.template`（任务意图表），再由项目负责人、规划型 AI、CLI 或插件生成 CommandPack。

一个合格的 CommandPack 至少包括：

- planId / iteration
- repoRoot
- 当前分支
- 任务目标
- 必读上下文
- 允许修改路径
- 禁止修改路径
- 具体任务步骤
- UTF-8 要求
- anti-spaghetti 要求
- 必跑门禁
- 验收标准
- ExecutionReport 格式
- 回滚要求
- STOP 条件

AI 执行越稳定，靠的不是“多提醒几句”，而是任务合同足够清楚。

## 16. 每次 AI 执行完成后，人要做什么

AI 返回 ExecutionReport 后，人不能只看“完成了”三个字。

应该检查：

1. 修改文件是否在允许范围内？
2. 是否改了无关文件？
3. 是否新增了不该新增的依赖？
4. 是否修改了业务规则？
5. 是否把逻辑塞进中央大文件？
6. 是否有中文乱码？
7. 是否真的运行了门禁？
8. 失败的门禁有没有解释？
9. 回滚方式是否清楚？
10. 下一步建议是否合理？

如果 AI 没跑测试，却说“应该可以”，不能算完成。

没有证据，就不要收口。

---

## 17. 收口规则：每个阶段结束要更新文档

每完成一个重要 PR，不要只提交代码。

还要更新至少一个上下文文件。

常见更新：

- `docs/PR_SUMMARIES.md`
- `docs/CONTEXT_PACK.md`
- `docs/MODULE_BOUNDARY.md`
- `docs/TESTING.md`
- `docs/NEXT_PHASE_PLAN.md`

如果本次新增了模块，更新 `MODULE_BOUNDARY.md`。

如果本次新增了命令，更新 `TESTING.md`。

如果本次改变了项目状态，更新 `CONTEXT_PACK.md`。

如果本次完成一个阶段，更新 `PR_SUMMARIES.md`。

文档不更新，后续 AI 就会断上下文。

## 18. 项目推进节奏

### 阶段 1：骨架期

目标：

- 跑起来。
- 文档立起来。
- 模块边界立起来。
- Git 基线建立。
- 基础门禁建立。

不要急着堆功能。

### 阶段 2：核心闭环期

目标：

- 完成最小可用功能链路。
- 数据结构稳定。
- 核心模块稳定。
- 基础测试跑通。

不要过早做复杂 UI。

### 阶段 3：能力扩展期

目标：

- 一个能力一个 PR。
- 每个能力有明确入口。
- 每个能力有验证方式。
- 新逻辑优先模块化。

不要多个功能混在一个 PR。

### 阶段 4：稳定化期

目标：

- 修 bug。
- 补测试。
- 梳理文档。
- 做回归。
- 准备打包或发布。

不要在稳定化期继续乱加功能。

### 阶段 5：发布收口期

目标：

- 打 tag。
- 做备份。
- 生成发行说明。
- 检查 license / 配置 / 构建产物。
- 明确回滚方案。

发布前不要做大重构。

## 19. 三种任务模式怎么选

### FAST MODE

适合：

- 小 bug。
- 小文案。
- 小样式。
- 小类型错误。
- 单文件低风险修复。

要求：

- 快。
- 小。
- 可验证。
- 不碰业务核心。

### SAFE MODE

适合：

- 普通功能。
- 普通 PR。
- 小模块开发。
- UI 与数据联动。
- 常规 bugfix。

要求：

- 读上下文。
- 锁范围。
- 跑门禁。
- 交报告。

### AUDIT MODE

适合：

- 架构调整。
- 重构。
- license。
- 权限。
- 计费。
- schema。
- 生产配置。
- 发布前检查。
- 仓库收口。

要求：

- 上下文完整。
- 计划清楚。
- 风险清楚。
- 门禁完整。
- 回滚清楚。

默认用 SAFE MODE。  
小任务可用 FAST。  
碰核心规则必须 AUDIT。

## 20. 不要把 AI 当成项目记忆

AI 可以辅助理解，但项目记忆应该沉淀在仓库里。

长期规则应该写入：

- `AGENTS.md`
- `docs/CONTEXT_PACK.md`
- `docs/MODULE_BOUNDARY.md`
- `docs/TESTING.md`
- `docs/PR_SUMMARIES.md`

不要依赖某一次聊天记录。

聊天会断，模型会换，上下文会丢。  
仓库文档才是长期协作的锚点。

## 21. 人的职责

项目负责人要做的不是亲自写所有代码，而是做好四件事。

### 1. 定方向

明确：

- 做什么。
- 不做什么。
- 先做什么。
- 后做什么。

### 2. 定边界

明确：

- 哪些文件能改。
- 哪些文件不能改。
- 哪些模块负责什么。
- 哪些规则不能碰。

### 3. 定验收

明确：

- 怎么算完成。
- 跑哪些测试。
- 哪些失败不能接受。
- 有无回滚方案。

### 4. 做审查

检查：

- AI 有没有越界。
- 有没有乱码。
- 有没有屎山化。
- 有没有假装成功。
- 有没有漏掉风险。

人不是要和 AI 比写代码速度。  
人要做的是守住方向、结构、边界和质量。

---

## 22. 常见错误

### 错误 1：一上来就让 AI 写功能

结果：

- 没有结构。
- 没有边界。
- 没有测试。
- 后面越来越乱。

正确做法：

先建文档、立边界、建 Git 基线，再写功能。

### 错误 2：一个任务塞太多目标

例如：

> 顺便优化 UI，顺便修 bug，顺便重构一下，顺便接入 license。

结果：

- diff 巨大。
- 不可审查。
- 不可回滚。
- 出错不知道是哪一步。

正确做法：

一个 PR 一个意图。

### 错误 3：允许路径太窄

例如只允许 AI 改 `main.js`，但任务实际需要新增模块。

结果：

- AI 把新逻辑塞进大文件。
- 文件越来越难维护。

正确做法：

如果任务需要新能力，就允许新增合理模块路径。

### 错误 4：不看 ExecutionReport

AI 说完成，不代表真的完成。

必须看：

- 改了什么。
- 跑了什么。
- 失败了什么。
- 风险是什么。
- 怎么回滚。

### 错误 5：文档不更新

代码变了，文档没变。  
下次 AI 读取旧文档，就会做错。

正确做法：

重要 PR 后必须更新上下文文档。

## 23. 最小可行项目启动流程

一个新项目最小启动流程：

1. 创建仓库。
2. 写 `README.md`。
3. 写 `AGENTS.md`。
4. 写 `docs/PRODUCT_BRIEF.md`。
5. 写 `docs/MODULE_BOUNDARY.md`。
6. 写 `docs/TESTING.md`。
7. 添加 `.editorconfig`。
8. 添加 `.gitattributes`。
9. 添加 `.gitignore`。
10. 添加 `scripts/check_utf8.py`。
11. 初始化 Git。
12. 创建 baseline commit。
13. 再开始第一个功能 PR。

这套流程看起来慢，但后面会省大量返工时间。

---

## 24. 每次迭代标准流程

每次迭代按这个流程：

1. 人先明确目标。
2. 判断任务模式：FAST / SAFE / AUDIT。
3. 生成 CommandPack。
4. AI 读取上下文。
5. AI 执行最小修改。
6. AI 跑门禁。
7. AI 输出 ExecutionReport。
8. 人审查 diff 和报告。
9. 通过后提交。
10. 更新 PR_SUMMARIES / CONTEXT_PACK。
11. 进入下一轮。

不要跳过第 8 步。  
不要跳过第 10 步。

---

## 25. 最终目标

这套流程的目标不是让开发变慢，而是让项目长期变快。

真正拖慢项目的不是前期多写几份文档，而是后期：

- AI 乱改。
- 上下文断裂。
- 中文乱码。
- 代码屎山。
- 测试缺失。
- 回滚困难。
- 每次都要重新解释项目。
- 修一个 bug 引出三个新 bug。

好的 AI 编程协作，不是 vibe coding。

它应该是：

> 有上下文、有范围、有边界、有门禁、有回滚、有报告的工程协作流程。
