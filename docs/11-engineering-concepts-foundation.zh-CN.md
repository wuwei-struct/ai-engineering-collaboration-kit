# AI 编程工程概念基础

> 本文件不是编程语言教程，也不是框架教程。
>
> 它是一份给 AI 编程用户看的工程概念地图，用来解释在 CommandPack、ContextPack、门禁、部署、权限、安全、数据和交付过程中经常出现的基础概念。
>
> 目标是帮助用户知道：
>
> - 这些词是什么意思；
> - 为什么 AI 编程中经常遇到；
> - 哪些概念属于高风险；
> - 什么时候应该让 AI STOP，而不是继续改。

---

## 1. 为什么需要工程概念基础

AI 可以写代码，但项目交付依赖工程概念。

如果用户不理解门禁、依赖、权限、幂等、schema、migration、回滚等基本概念，就很难判断：

- AI 当前在做什么。
- AI 是否越界。
- 某次修改会不会造成严重后果。
- 什么时候该说 STOP。

这份文档帮助用户理解：

- CommandPack 为什么需要门禁、回滚、路径、风险、验收标准。
- ContextPack 为什么需要分层和阶段性整理。
- 哪些概念是低风险的，哪些概念一旦碰错就可能出事故。

---

## 2. 最小必懂概念

以下是普通用户至少需要知道的 10 个工程概念：

### 项目

一个完整的软件工程产物，包含代码、文档、配置、测试、资源等。不是单个文件，而是一组有组织的文件集合。

### 文件 / 目录

代码、文档、配置以文件形式存放。目录（文件夹）用于组织文件。项目结构决定了代码好不好找、好不好改。

### 模块

项目中的一个独立功能区域，通常对应一个目录。模块之间可以互相调用，但要遵守边界规则。好的模块设计让改 A 不影响 B。

### 入口文件

项目的启动文件，例如 `main.js`、`index.js`、`App.tsx`。入口文件只负责初始化和接线，不应该塞满业务逻辑。

### 依赖

项目运行需要的外部代码，例如第三方库、框架、工具。依赖的版本变更可能破坏现有功能。AI 改代码时，不应随意升级依赖。

### 配置

控制项目行为的设置，例如端口号、数据库地址、API 地址、开关。配置和代码应该分离。生产配置尤其不能随便改。

### 环境变量

在系统级别传递的配置值，例如 `DATABASE_URL`、`API_KEY`。环境变量很适合存放不同环境（开发、测试、生产）之间的差异信息。绝不能把密钥硬编码在代码里。

### 测试

验证代码是否按预期工作的自动化检查，包括单元测试、集成测试、类型检查等。没有测试证据的情况下，不要声称"完成了"。

### 构建

将源代码转换成可运行产物的过程，例如编译、打包、压缩。构建失败通常意味着代码有严重问题。

### 回滚

将代码恢复到之前的某个稳定版本。每次重要修改前应该确保有办法回滚。没有回滚方案的操作是高风险的。

## 3. CommandPack 相关概念

CommandPack 是 AI 执行本次任务的"合同"。理解以下概念有助于正确使用 CommandPack：

### 任务目标

本次要完成的具体内容。必须是明确、可验证的。模糊的目标会导致 AI 自由发挥、扩大范围。

### 非目标

明确说明本次**不做什么**。防止 AI 顺手做超出范围的事情。

### 允许修改路径

AI 可以改哪些文件或目录。明确的路径范围防止 AI 改到不该改的地方。

### 禁止修改路径

AI 不能改哪些文件或目录。例如技能规则、脚本、许可证、模板。这相当于给 AI 画了一道"不许过"的线。

### 必读上下文

AI 在执行任务前必须阅读哪些文件。确保 AI 在正确理解项目现状的基础上行动。

### 验收标准

任务是否完成的具体判断条件。例如"所有测试通过""UTF-8 检查通过""特定 API 返回正确数据"。没有验收标准的任务无法判断是否真正完成。

### STOP 条件

遇到什么情况 AI 必须停止，不能继续修改。例如工作区不干净、tag 已存在、需要修改禁止路径、涉及密钥或生产配置。STOP 不是失败，是保护。

### ExecutionReport

AI 完成任务后输出的执行报告，记录修改了哪些文件、通过了哪些门禁、有哪些风险、如何回滚。ExecutionReport 是人的审查入口。

### 回滚要求

任务说明中关于如何恢复到修改前状态的要求。包括 baseline commit、tag、rollback notes 等。

### 文档回写

任务完成后是否需要更新项目文档（如 ContextPack、PR_SUMMARIES、CHANGELOG）。重要变更后必须回写文档，否则项目文档会越来越滞后。

## 4. ContextPack 相关概念

ContextPack 是项目的"长期记忆包"，让 AI 在多次对话之间保持对项目的理解。

### 上下文

AI 理解当前项目所需的信息，包括项目定位、模块结构、当前状态、最近完成的任务、禁止事项等。没有上下文，AI 只能靠猜测。

### ContextPack

压缩后的项目上下文文档。通常放在 `docs/CONTEXT_PACK.md`。它不等于完整项目文档，而是给 AI 快速读取的关键信息摘要。

### L1 / L2 / L3

ContextPack 的三个层次：

- **L1**：项目基本情况，给首次接触项目的 AI 快速了解。
- **L2**：中等详细程度，包含模块说明、关键决策、当前重点。
- **L3**：详细上下文，用于审计、交接、深度分析。

大多数日常任务只需要 L1 或 L2。

### Context Delta

任务完成后，项目上下文发生了变化，这个"变化的部分"就是 Context Delta。每次重要任务后应该记录 Delta，便于后续整理 ContextPack。

### 文档回写

将 Context Delta 同步回项目文档（CONTEXT_PACK.md、PR_SUMMARIES.md 等）。如果不回写，ContextPack 会越来越脱离实际。

### PR_SUMMARIES

迭代记录文档，记录每次重要任务的"做了什么、为什么做、影响什么"。这是项目的长期协作记忆。

### 任务历史

完成的任务列表和关键决策记录。帮助理解项目走了多远、为什么做了某些选择。

### 项目当前状态

项目目前处在什么阶段：哪些功能完成了、哪些在进行中、哪些暂时不做。AI 不应该重复提出已经否定过的方案。

## 5. 代码结构概念

### 前端

用户能看到和交互的部分，通常在浏览器里运行。例如网页、手机 App 界面、桌面应用界面。

### 后端

在服务器上运行的逻辑，用户看不到但结果由它计算。例如数据处理、用户认证、业务规则。

### API

应用程序之间的通信接口。前端通过 API 向后端请求数据或执行操作。API 的输入输出格式变更会影响所有调用者。

### IPC

进程间通信（Inter-Process Communication）。在桌面应用中，主进程和渲染进程之间的通信方式。IPC 合约变更必须谨慎。

### service

服务层，包含核心业务逻辑。例如 `users.service.js` 负责用户相关的所有业务操作。不要把业务逻辑堆在入口文件里。

### adapter

适配器，把外部系统或旧接口转换成项目内部统一格式。例如把不同支付平台的接口统一成内部格式。

### parser

解析器，把某种格式的数据转换成另一种格式。例如把 JSON 字符串转成对象，把 Markdown 转成 HTML。

### builder

构建器，按步骤构建复杂对象。例如构建数据库查询语句、构建 HTML 页面。

### schema

数据结构定义，描述数据的形状、类型、约束。例如"用户必须有 name（字符串）和 email（邮箱格式）"。schema 变更通常会触发 migration。

### store

状态存储，管理应用程序的数据状态。例如 React 的 Redux store、Vue 的 Pinia store。状态结构变更会影响所有使用该状态的组件。

### component

组件，UI 的独立可复用单元。例如一个按钮、一个表单、一个列表。组件之间的接口变更要谨慎。

### view

视图，用户看到的页面或页面区域。视图通常由多个组件组合而成。

### utility / utils

工具函数，通用的、与业务无关的辅助代码。例如日期格式化、字符串处理、数学计算。工具函数要简单、无副作用。

> 新逻辑不应该都塞进入口文件。
> 不懂模块边界时，不要让 AI 大重构。

## 6. 操作与协作概念

### Git

版本控制系统，用于追踪代码变更、协作开发、版本管理。几乎所有现代软件项目都使用 Git。

### commit

一次代码保存操作，记录"谁在什么时候改了什么"。每个 commit 应该做一件事，便于后续追踪和回滚。

### branch

分支，代码的独立开发线。新功能在分支上开发，完成后合并回主线。分支隔离了开发中的不稳定性。

### tag

标签，给某个 commit 打一个固定名称，例如 `v0.1.10`。标签用于标记发布版本和关键节点。**永远不要覆盖已有 tag。**

### diff

代码差异，显示两个版本之间的变化。`git diff` 是审查 AI 修改内容的基本工具。

### PR

Pull Request，合并请求。开发者在分支上完成工作后，通过 PR 请求将代码合并到主线。PR 是代码审查的入口。

### merge

合并，把一个分支的代码并入另一个分支。合并不当会导致冲突。

### rollback

回滚，把代码恢复到之前的某个版本。有 baseline commit 或 tag 时，回滚非常安全。

### baseline

基线，一个稳定、已验证的代码版本。所有后续修改都从基线出发。出问题时可以回归基线。

### changelog

变更日志，记录每个版本的新增、修改、修复内容。CHANGELOG 是给人和工具看的版本历史。

## 7. 测试与门禁概念

### lint

代码风格检查，发现格式问题、潜在错误、不符合规范的写法。lint 不通过意味着代码质量不达标。

### typecheck

类型检查，验证变量和函数的类型是否匹配。类型错误可能导致运行时崩溃。

### unit test

单元测试，验证单个函数或模块的行为是否符合预期。单元测试是最基础的门禁。

### integration test

集成测试，验证多个模块组合在一起时是否正常工作。集成测试捕获模块之间的不兼容。

### smoke test

冒烟测试，快速验证核心功能是否正常工作。通常在构建后或部署前运行。

### build

构建，把源码转换成可运行产物。构建失败意味着代码有无法运行的严重问题。

### CI

持续集成（Continuous Integration），自动运行 lint、typecheck、test、build 的系统。每次提交代码时 CI 自动检查。

### gate / 门禁

必须通过才能继续的条件。例如"lint 和 test 全部通过才允许合并"。门禁是质量的最后防线。

### fixture

测试夹具，预先准备好的测试数据。保证测试可重复、结果可预测。

### mock

模拟对象，用来替代真实的外部依赖（如数据库、API）。让测试在不连接真实外部系统的情况下运行。

### real test

真实测试，使用真实的数据库、真实的服务进行测试。比 mock 测试更接近生产环境，但更慢更复杂。

> 没有验证证据，不算完成。

## 8. 高风险工程概念

本节是本文档的重点。以下概念在日常开发中风险很高，AI 修改时尤其需要谨慎。

### 权限

**是什么**：系统控制"谁能做什么"的机制。例如"管理员可以删除用户，普通用户只能查看"。

**为什么高风险**：

- 权限漏洞可能导致数据泄露或越权操作。
- 权限逻辑通常分布在多个模块中，改一处可能影响全局。
- AI 可能无意中放宽或收紧权限，而用户看不出。

**AI 修改时**：不应直接修改权限逻辑。涉及权限的改动必须由了解业务规则的人审查。

**什么时候 STOP**：当 AI 试图修改权限检查、角色定义、访问控制时，应 STOP。

---

### 认证 / auth

**是什么**：验证用户身份的系统，例如登录、token 验证、OAuth。

**为什么高风险**：

- 认证漏洞可能导致账号被盗、未授权访问。
- 认证逻辑涉及加密、token、session 等安全敏感操作。
- AI 可能引入不安全的认证方式或破坏现有安全机制。

**AI 修改时**：不建议直接修改认证逻辑。涉及认证的改动必须由安全审查。

**什么时候 STOP**：当 AI 试图修改登录流程、token 处理、session 管理时，应 STOP。

---

### license / 激活码

**是什么**：控制软件使用权限的许可证系统，包括激活码生成、验证、过期管理。

**为什么高风险**：

- license 逻辑直接关联商业模式和收入。
- license 漏洞可能导致收入损失。
- license 验证涉及加密和防篡改，改错会导致合法用户无法使用。

**AI 修改时**：不建议直接修改 license 相关逻辑。

**什么时候 STOP**：当 AI 试图修改激活码验证、许可证检查、试用期逻辑时，应 STOP。

---

### 计费 / billing

**是什么**：计算和收取用户费用的系统，包括定价、计费周期、扣款逻辑。

**为什么高风险**：

- 计费错误直接导致经济损失或用户投诉。
- 涉及真实货币，不能有任何偏差。
- 计费逻辑通常很复杂，与业务规则深度耦合。

**AI 修改时**：不建议直接修改计费逻辑。涉及计费的改动必须经过严格的业务验证。

**什么时候 STOP**：当 AI 试图修改扣款逻辑、定价规则、计费周期时，应 STOP。

---

### 支付

**是什么**：处理真实货币交易的系统，通常对接第三方支付平台。

**为什么高风险**：

- 涉及真实资金，错误不可接受。
- 支付接口有严格的安全和合规要求。
- 支付回调处理不当可能导致重复扣款或漏单。

**AI 修改时**：不建议直接修改支付相关逻辑。

**什么时候 STOP**：当 AI 试图修改支付流程、支付回调、金额计算时，应 STOP。

---

### 用户数据

**是什么**：用户在使用产品过程中产生或提供的数据，包括个人信息、使用记录、内容等。

**为什么高风险**：

- 用户数据涉及隐私和合规（如 GDPR、个人信息保护法）。
- 数据泄露可能导致法律风险。
- AI 可能无意中暴露或错误处理用户数据。

**AI 修改时**：涉及用户数据的读/写/传输/存储时需要谨慎。不要将真实用户数据交给 AI 处理。

**什么时候 STOP**：当任务涉及批量处理用户数据、修改数据访问权限、改变数据存储方式时，应 STOP。

---

### 数据库

**是什么**：持久化存储数据的系统，例如 MySQL、PostgreSQL、MongoDB。

**为什么高风险**：

- 数据库存储核心业务数据，操作不可逆。
- 错误的查询或修改可能导致数据丢失。
- AI 可能写出低效或危险的数据库操作。

**AI 修改时**：涉及数据删除、批量更新、结构变更时需要验证。不应在生产数据库上直接执行 AI 生成的 SQL。

**什么时候 STOP**：当 AI 试图直接操作生产数据库、执行删除或批量修改时，应 STOP。

---

### schema

**是什么**：数据库或数据结构的定义，描述数据"长什么样"。例如"users 表有 id、name、email 三个字段"。

**为什么高风险**：

- schema 变更会影响所有读写该数据的代码。
- schema 变更不可轻易回滚（数据可能已经改变）。
- 错误的 schema 变更可能导致数据不兼容。

**AI 修改时**：schema 变更通常需要配合 migration，两者都要谨慎。

**什么时候 STOP**：当 AI 试图删除字段、更改字段类型、修改约束时，应 STOP 并确认。

---

### migration

**是什么**：数据库迁移，增量式地修改数据库结构以匹配新的 schema。例如"给 users 表增加 phone 字段"。

**为什么高风险**：

- migration 直接修改数据库结构，操作不可逆或难以回滚。
- 错误的 migration 可能破坏现有数据或导致服务不可用。
- 在生产环境执行 migration 之前必须有充分的测试。

**AI 修改时**：不应在生产环境直接执行 AI 生成的 migration。migration 必须经过代码审查和测试环境验证。

**什么时候 STOP**：当 AI 试图创建或修改 migration，且未明确测试环境验证步骤时，应 STOP。

---

### 幂等

**是什么**：同一个操作执行一次和执行多次，结果应该保持一致。

例如：用户点击"扣点"按钮一次扣 1 点，重复点击或请求重试时不能扣多次。

**为什么高风险**：

- 网络请求可能重复。
- 用户可能重复点击。
- AI 修改接口时可能破坏防重复逻辑。
- 计费、支付、激活码、积分系统都很依赖幂等。

**AI 修改时**：遇到幂等相关逻辑，应进入高风险判断，不应随意改。

**什么时候 STOP**：当 AI 试图修改涉及扣款、扣点、激活、计费的接口，且未确认幂等保护是否完好的情况下，应 STOP。

---

### 并发

**是什么**：多个操作同时发生时，系统如何保证数据一致性。

例如：两个用户同时购买最后一件库存商品，系统必须保证只卖出一次。

**为什么高风险**：

- 并发问题难以重现和测试。
- 错误的并发处理可能导致数据错乱。
- 并发 bug 通常在真实使用场景下才会暴露。

**AI 修改时**：涉及共享状态、库存、余额、计数器的操作必须考虑并发安全。

**什么时候 STOP**：当 AI 试图修改涉及共享资源（库存、余额、计数器）的逻辑，但未说明并发安全措施时，应 STOP。

---

### 事务

**是什么**：一组操作要么全部成功，要么全部撤销。保证数据在多次操作过程中保持一致。

例如：转账操作中，"A 账户扣钱"和"B 账户加钱"必须同时成功或同时失败，不能只完成一半。

**为什么高风险**：

- 事务保证了数据的原子性和一致性。
- 错误的事务处理可能导致数据不一致。
- 在生产环境中修复事务错误非常困难。

**AI 修改时**：涉及多步数据操作的逻辑，需要确认事务边界是否合理。

**什么时候 STOP**：当 AI 试图修改涉及多表、多步写操作的逻辑，但未考虑事务时，应 STOP。

---

### 缓存

**是什么**：临时存储计算结果，下次请求直接返回缓存结果，不用重新计算。

例如：首页排行榜每小时更新一次，在两次更新之间用缓存结果。

**为什么高风险**：

- 缓存过时会导致用户看到旧数据。
- 缓存策略不当可能导致数据不一致。
- AI 可能忽略缓存失效逻辑，导致 bug。

**AI 修改时**：涉及缓存策略、缓存失效、缓存键的修改需要验证数据一致性。

**什么时候 STOP**：当 AI 试图修改缓存逻辑，且未说明缓存失效策略时，应 STOP。

---

### 队列

**是什么**：异步处理任务的机制。任务先放入队列，后台逐个处理。

例如：用户上传图片后，图片压缩任务放入队列，后台慢慢处理，用户不用等待。

**为什么高风险**：

- 队列任务失败可能导致数据丢失。
- 队列积压可能导致系统卡住。
- 队列中的任务需要幂等处理（可能被重复投递）。

**AI 修改时**：涉及队列生产者/消费者的修改需要验证失败重试和幂等。

**什么时候 STOP**：当 AI 试图修改队列任务格式、消费者逻辑，但未考虑向后兼容时，应 STOP。

---

### 生产配置

**是什么**：线上运行环境的配置，例如生产数据库地址、生产 API 密钥、生产端口号。

**为什么高风险**：

- 错误的生产配置可能导致服务不可用。
- 修改生产配置没有"回滚"缓冲，直接影响线上用户。
- 生产配置通常不应由 AI 直接修改。

**AI 修改时**：AI 不可直接修改生产配置。涉及生产配置的改动必须人工执行。

**什么时候 STOP**：当 AI 试图修改任何生产环境配置时，必须 STOP。

---

### 密钥 / token / API Key

**是什么**：用于身份验证和授权的秘密凭证。例如 GitHub token、AWS API Key、数据库密码。

**为什么高风险**：

- 密钥泄露等于授权泄露。
- 密钥一旦暴露在代码或日志中，必须立即作废。
- AI 不应读取或处理真实密钥。

**AI 修改时**：不可读取或修改密钥。密钥不应出现在代码、日志、配置文件中（应放在环境变量或密钥管理服务中）。

**什么时候 STOP**：当 AI 试图读取、生成、修改或打印任何密钥时，必须 STOP。

---

## 9. 部署上线概念

### 本地环境

开发者自己的电脑，用于编写和初步测试代码。AI 在本地环境执行任务是安全的。

### 开发环境

团队共享的测试服务器，用于集成测试和联调。

### 测试环境

接近生产环境的独立环境，用于上线前的最终验证。

### 生产环境

用户正在使用的线上环境。**AI 绝不能直接操作生产环境。**

### server

服务器，运行后端代码的计算机。可以是物理机、虚拟机或云服务。

### domain

域名，用户访问服务的地址，例如 `example.com`。域名配置错误会导致服务无法访问。

### SSL

安全套接层，为网站提供 HTTPS 加密。SSL 证书过期会导致用户无法安全访问。

### database

见上文"高风险工程概念"中的数据库部分。

### migration

见上文"高风险工程概念"中的 migration 部分。

### environment variables

见上文"最小必懂概念"中的环境变量部分。不同环境的变量值不同，但不应改变变量的"存在性"。

### log

日志，系统运行记录。日志帮助排查问题，但不能包含密钥、密码等敏感信息。

### backup

备份，数据的定期快照。没有备份就没有真正意义上的安全。涉及生产数据修改前必须确认有备份。

### rollback

见上文"操作与协作概念"中的回滚部分。部署上线必须有明确的回滚方案。

### release

发布，将新版本部署到生产环境。发布前必须通过所有门禁，并有回滚方案。

> 生产配置不能随便改。
> 上线前必须有回滚方案。

## 10. AI 协作相关概念

### vibe coding

一种非正式的 AI 编程风格：用户用自然语言描述想要的，AI 持续生成代码。适合快速原型和探索，但缺乏工程纪律。长期项目不能只靠 vibe coding。

### Skill

AI 执行代理使用的压缩执行规则。它告诉 AI 什么时候 FAST、SAFE、AUDIT，什么情况必须 STOP，如何输出 ExecutionReport。Skill 是 AI 的"工作规范"。

### CommandPack

AI 本次任务的"合同"，包含任务目标、允许/禁止路径、必读上下文、验收标准、STOP 条件。类似工人的施工许可证。

### Task Intake

普通用户的任务意图表，用自然语言描述"想做什么"。不需要用户懂工程术语。上游规划者据此生成 CommandPack。

### Project Intake

项目初始化意图表，收集项目基本情况。可用于生成初始 ContextPack。

### ContextPack

项目的"长期记忆包"，让 AI 在多次对话之间保持对项目的理解。见上文 ContextPack 相关概念。

### ExecutionReport

AI 完成任务后必须输出的执行报告。记录修改了什么、通过了哪些门禁、有哪些风险、如何回滚。

### Context Writeback

将任务完成后的上下文变化同步回项目文档。确保 ContextPack 不落后于实际代码。

### 受控顺手修

AI 在低风险、同因、局部、可验证的条件下，可以处理必要的同类小修复。但不是"想改什么就改什么"。涉及高风险概念时必须 STOP。

### STOP

AI 必须停止执行的信号。STOP 不是失败，是保护——保护代码、保护数据、保护用户。当 AI 遇到权限、license、计费、密钥、生产配置、schema、migration 等高风险区域，或规则不明确时，应当 STOP。

### 任务合同库

一组已完成的 CommandPack 集合。可用于复盘、审计和生成新的 CommandPack。

## 11. 概念风险等级表

| 概念 | 风险等级 | AI 可否直接改 | 建议 |
|---|---|---|---|
| 文案 / 提示文本 | 低 | 可以，按范围 | FAST |
| 样式 / 排版 | 低 | 可以，按范围 | FAST |
| 普通组件 / 页面 | 中 | 可以，但要验证 | SAFE |
| 工具函数 / utils | 中 | 可以，但要测试 | SAFE |
| 入口文件 | 中 | 谨慎，不能塞逻辑 | AUDIT |
| 模块边界 | 中 | 谨慎，需确认设计 | AUDIT |
| API contract / 接口格式 | 高 | 谨慎 | AUDIT / STOP |
| 依赖升级 | 高 | 不建议直接改 | STOP |
| 权限 | 高 | 不建议直接改 | STOP |
| 认证 / auth | 高 | 不建议直接改 | STOP |
| license / 激活码 | 高 | 不建议直接改 | STOP |
| 计费 / billing | 高 | 不建议直接改 | STOP |
| 支付 | 高 | 不建议直接改 | STOP |
| 用户数据 | 高 | 不建议直接改 | STOP |
| schema / migration | 高 | 不建议直接改 | STOP |
| 幂等逻辑 | 高 | 不建议直接改 | STOP |
| 并发逻辑 | 高 | 不建议直接改 | STOP |
| 事务逻辑 | 高 | 不建议直接改 | STOP |
| 缓存策略 | 高 | 不建议直接改 | STOP |
| 队列逻辑 | 高 | 不建议直接改 | STOP |
| 生产配置 | 极高 | 不可直接改 | STOP |
| 密钥 / token | 极高 | 不可读取或修改 | STOP |
| 数据库删除/批量更新 | 极高 | 不可直接执行 | STOP |

## 12. 普通用户怎么用这份文档

- **不需要一次学完。** 这是一份概念地图，不是考试教材。
- **遇到 CommandPack 或 ContextPack 里不懂的词，回来查。**
- **遇到"高风险"概念，要谨慎。** 不是说你不能提出需求，而是说你需要在需求中清楚说明"这是计费相关，请谨慎"。
- **不懂就让 AI 解释，不要让 AI 直接改。** AI 可以帮你解释"什么是幂等"，但在你理解之前，不要让 AI 动幂等相关的代码。
- **判断不了风险等级时，让 AI STOP 并询问项目负责人。**

## 13. 一句话总结

> AI 编程不是只会描述需求就够了。
> 想做出可交付项目，至少要知道哪些东西是低风险，哪些东西一碰就可能出事故。
