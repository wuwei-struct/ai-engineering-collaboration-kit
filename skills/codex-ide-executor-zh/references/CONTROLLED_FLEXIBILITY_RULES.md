# 受控灵活性规则

默认不得扩大任务范围。  
但允许在极低风险条件下处理同类小修复。

---

## 1. 必须满足

- 位于本次允许修改路径内。
- 与当前任务属于同一问题、同一模式或同一直接原因。
- 改动很小。
- 不改变 API contract。
- 不改变 schema。
- 不改变权限规则。
- 不改变 license / 计费逻辑。
- 不改变用户数据结构。
- 不改变导出格式。
- 不改变生产配置。
- 不引入新依赖。
- 不触碰密钥。
- 不造成中央大文件继续膨胀。
- 可以被当前门禁或相关测试覆盖。
- 在 ExecutionReport 中明确列出。

---

## 2. 量化边界

通常应满足：

- 单文件内顺手修复不超过约 20 行。
- 跨文件顺手修复不超过 3 个文件。
- 跨文件总改动不超过约 30 行。
- 不新增公共 API。
- 不新增新的函数签名。
- 不新增新的导出。
- 不新增新的业务分支。
- 不新增新的依赖。
- 不新增新的数据结构。
- 不新增新的配置项。
- 不改变现有调用契约。
- 不改变测试策略。
- 不改变用户可见功能边界。

超过阈值时，STOP 并建议拆分任务。

---

## 3. 当前处理 vs 下一步建议

| 问题严重度 | 改动范围 | 与当前任务关系 | 处理方式 |
|---|---|---|---|
| 高 | 小 | 同因 | 当前处理或 STOP，视是否涉及硬规则 |
| 高 | 大 | 同因 | STOP，拆任务 |
| 中 | 小 | 同因 | 可当前处理 |
| 中 | 中 | 同因 | 谨慎，超阈值 STOP |
| 低 | 小 | 无关 | 下一步建议 |
| 任意 | 大 | 任意 | STOP 或拆分 |
| 任意 | 权限/license/计费/schema/生产配置/密钥 | 任意 | STOP |
